Lab Set-up / Virtualization

Virtualization is the process of creating a virtual (software-based) version of a physical resource such as a server, operating system, storage device, or network. Instead of relying on dedicated hardware, virtualization allows multiple virtual instances to run on a single physical machine using a hypervisor (software that manages virtual machines).

Detailed Scenarios and Troubleshooting Procedures

1. Blue Screen of Death (BSOD) Errors

Scenario: A high-performance workstation crashes during data processing.

Step 1: Collect Crash Information
  • When BSOD appears, note the STOP Code (e.g., `0x00000133`).
  • After reboot, locate the minidump file: `C:\Windows\Minidump`
  • Copy the latest `.dmp` file to a secure administrative directory for analysis.
Step 2: Analyze Dump File
  • Install WinDbg (Windows Debugger).
  • Open the `.dmp` file: File → Open Crash Dump.
  • Execute the command to analyze the crash: `!analyze -v`
  • Identify the FAULTING_MODULE or IMAGE_NAME (e.g., `nvlddmkm.sys` indicating NVIDIA driver).
Step 3: Remediation Procedure
  • Boot workstation into Safe Mode.
  • Open Device Manager: `devmgmt.msc`
  • Locate the device associated with the faulty driver.
  • Right-click → Uninstall Device, check Delete the driver software for this device.
  • Download and install the latest driver from the manufacturer’s official site.
  • Reboot workstation and verify stability by executing the previously failing workload.

Outcome: BSOD issue is resolved, workstation stability is restored, and the correct driver version is deployed.

2. Network Connection Issues (LAN, WAN)

Scenario: A scientist cannot access the local data server (LAN) or ICAR-hosted external resources (WAN)

Step 1: Client-Side Diagnostic Checks
  • Verify IP/DNS: `ipconfig /all` (Verify IP, subnet mask, gateway, DNS)
  • LAN connectivity test: `ping [Server_IP]`
  • Internet/WAN test: `ping 8.8.8.8`
  • DNS resolution test: `nslookup www.icar.org.in`
  • Document the results for further troubleshooting.
Step 2: LAN Troubleshooting
  • Inspect physical connections: switch port LEDs, patch cables.
  • For managed switches: log into console or web interface and verify: `show interfaces status` and `show port-security`
  • Ensure the client’s MAC address is not blocked by port security rules.
Step 3: WAN/DNS Troubleshooting
  • Log into firewall/gateway: check logs for blocked outbound traffic.
  • Verify DNS server status: `Get-Service DNS` and `Test-DnsServer -Name www.icar.org.in`
  • Adjust firewall or DNS settings as required.

Outcome: Network connectivity restored for both LAN and WAN. Root cause documented and preventive measures applied.

3. Email Configuration and Troubleshooting (Outlook, Zoho Mail)

Scenario: An Admin Officer's Outlook client cannot send mail from home; receives an SMTP error.

Step 1: Verification and Isolation
  • Log into Zoho Webmail to confirm sending/receiving functionality (If webmail works, issue is client-side configuration or network blocking).
Step 2: Check Outlook Account Settings
  • Open Outlook → File → Account Settings → Account Settings → Change → More Settings.
  • Navigate to Outgoing Server (SMTP) tab.
Step 3: SMTP Authentication & Port Configuration
  • Ensure My outgoing server (SMTP) requires authentication is enabled.
  • Configure Advanced tab: SMTP Port: `587` (preferred) or `465`, Encryption: `SSL/TLS`.
  • Test sending an email; verify error resolution.

Outcome: SMTP errors resolved, client can send emails securely from remote location.

4. Office 365 Administration (MS Office, MS Teams)

Scenario: A new research project team needs a collaborative space and access to the latest Office apps.

Step 1: License Verification

Step 2: Provision MS Teams

Step 3: Configure Team Permissions

Outcome: Team is provisioned with correct licenses, permissions, and collaborative workspace. Users can safely access Office apps and Teams channels.

5. User & Access Management (Enforced AAA Model)

Scenario: Enforce strong authentication, define access rights, and track all actions for sensitive data.

Step 1: Authentication (A1) – Verify Identity
  • AD Password Policy: Configure Group Policy (GPO) for the R&D OU (Min length: 12 chars, Complexity: Enabled, Age: 60 days, Lockout: 5 failed attempts).
  • Remote Access 2FA via RADIUS: Install/configure RADIUS Server. Enable Two-Factor Authentication (2FA) for VPN users (Integration with Microsoft/Google Authenticator).
  • Example: `Get-ADDefaultDomainPasswordPolicy`
Step 2: Authorization (A2) – Define Access Rights
  • Granular NTFS Permissions: Identify shared drives (e.g., E:\Research_R&D).
  • Assign role-based permissions to Security Groups (SGs): `SG_Research_Head` (Full Control), `SG_Research_Staff` (Modify), `SG_IT_Admin` (Full Control). Remove "Everyone" permission.
  • Example PowerShell: `Set-Acl "E:\Research_R&D" $acl`
Step 3: Auditing (A3) – Track User Actions
  • Enable Account and Object Auditing: Configure Advanced Audit Policy via GPO for Account Logon, Logon/Logoff, and Object Access (Success and Failure).
  • Configure audit policy to forward logs to central SIEM / Log Server.
  • Verify Auditing: Test successful/failed logins and access to restricted folders.
  • Example Log Check: `Get-WinEvent -LogName Security -MaxEvents 50 | Where-Object {$_.Id -in 4624,4625,4663}`

Outcome: Authentication ensures authorized login (with 2FA). Authorization provides granular, role-based access. Auditing tracks all critical actions for compliance and incident investigation.

6. Backup Strategies and Disaster Recovery (3-2-1 Rule & DR)

Scenario: Implement a resilient strategy for the critical research database server.

Step 1: Apply the 3-2-1 Backup Policy
  • Three Copies: Production → Local NAS → Off-site cloud.
  • Two Media Types: RAID-protected server disks → Tape / Cloud.
  • One Off-Site Copy: Cloud replication (AWS S3 / Azure Blob) with encryption.
  • Retention: Daily incrementals (7 days), weekly full (4 weeks), monthly archive (12 months).
  • Example Cloud Sync: `aws s3 sync E:\Research_DB s3://research-db-backup --storage-class STANDARD_IA --sse AES256`
Step 2: Configure Local Backup (e.g., Veeam)
  • Create Backup Job (e.g., `Research_DB_Full`). Add server and choose Application-Aware Processing for MS SQL.
  • Target repository: LocalNAS. Schedule: Daily 2 AM. Enable retries = 3.
  • Validate Backup: Perform a test restore of a single DB to ensure consistency.
  • Example status check: `Get-VBRBackupSession -Last 7`
Step 3: Configure Off-Site Replication & Disaster Recovery (DR) Drill
  • Backup Copy Job: Source: Local NAS → Destination: Cloud Gateway. Enable throttling.
  • DR Drill (Quarterly): Define RTO (≤4 hours), RPO (≤15 minutes). Isolate production network: `Set-NetAdapter -Name "Ethernet0" -AdminStatus Down`.
  • Execute Failover Drill: Boot VM from off-site backup (`Start-VM`). Test full system functionality.
  • Post-Drill: Document total failover time vs RTO and reconnect production network.
Step 4: Reporting and Audit
  • Generate backup and replication reports: `Get-VBRBackupSession -Last 7`
  • Maintain DR Drill Report with: Drill date & duration, VM boot time, Database and app verification, Lessons learned.

Outcome: Critical research database is fully protected under the 3-2-1 rule. Quarterly DR drills validate recovery procedures and meet RTO/RPO objectives.

7. System Configuration, Maintenance, and Patching

Scenario: Apply monthly security patches to all staff workstations and servers with minimal disruption.

Step 1: Create Test Group
  • Open WSUS (Windows Server Update Services) or SCCM.
  • Create a computer group named `IT_Patch_Test`.
  • Add a small number of non-critical workstations and test virtual machines (VMs).
  • Verify the devices are properly reporting.
Step 2: Approve and Deploy Test Updates
  • Approve the latest security updates only for the Test Group.
  • Set a deployment deadline of 48 hours.
  • Monitor deployment progress, check for failed updates, and address compatibility issues promptly.
Step 3: Production Deployment
  • Upon successful testing, approve updates for the remaining workstation groups.
  • Schedule server updates during maintenance windows (e.g., late Friday night).
  • Ensure full server backups are completed before patching begins.
  • Monitor the deployment for errors and confirm all systems are patched successfully.

Outcome: All workstations and servers are securely updated with minimal disruption. Testing ensures compatibility, reducing the risk of downtime or update-related issues.

8. File and Storage System Configuration

Scenario: Create a new shared drive for the R&D team with a space quota for responsible usage.

Step 1: Create Volume and Shared Folder
  • Open Server Manager → File and Storage Services. Create a New Volume on the server’s data disk.
  • Right-click the newly created volume → New Share Wizard.
  • Configure the share: Share Path: `E:\Research_R&D`, Share Name: `R&D_DATA`.
Step 2: Configure NTFS Permissions
  • Right-click the shared folder → Properties → Security Tab.
  • Add the `SG_RnD_Staff` security group. Assign appropriate permissions (e.g., Modify or Full Control).
  • Remove default Users or Everyone permissions to enforce access control.
  • Verify access by testing from a client workstation.
Step 3: Implement Storage Quota
  • Open File Server Resource Manager (FSRM) → Quota Management → Create Quota.
  • Select the path for the new shared folder.
  • Set quota properties (e.g., 500 GB Hard Limit).
  • Configure notifications if desired to alert users before reaching the quota. Apply and verify.

Outcome: The R&D team now has a dedicated, secure shared folder with controlled access and a defined storage limit, ensuring responsible usage and centralized management of server resources.

9. Set up and Maintained DNS and DHCP

Scenario: A new research instrument needs a static IP reservation, and the internal DNS needs a record for a new internal web portal.

Step 1: Configure Static DHCP Reservation
  • Open the DHCP Management Console. Navigate to the relevant scope.
  • Right-click Reservations → New Reservation.
  • Enter Reservation Name, MAC Address, and desired IP Address (e.g., `192.168.1.10`).
  • Verify the device receives the reserved IP on the next DHCP lease renewal.
Step 2: Create DNS Record
  • Open the DNS Management Console. Navigate to the Primary Forward Lookup Zone (e.g., `circot.icar.gov.in`).
  • Right-click New Host (A or AAAA) Record.
  • Enter Host Name: `portal` and IP Address: `[Web Server IP]`.
  • Optionally, check Create associated pointer (PTR) record for reverse lookup.
  • Test the DNS resolution using: `nslookup portal.circot.icar.gov.in`

Outcome: The research instrument consistently receives the designated IP address via DHCP, and the internal web portal can be accessed by its hostname, ensuring reliable network communication and name resolution within the organization.

10. Remote Administration (AnyDesk, TeamViewer)

Scenario: A specialized vendor needs temporary, monitored access to the Hyper-V host server for a software upgrade.

Step 1: Establish Security Perimeter
  • Obtain written administrative approval (Server name, Date, Purpose).
  • Create a dedicated administrative account for the vendor.
  • Assign temporary privileges; ensure the account will be disabled immediately after the session.
Step 2: Grant and Monitor Access
  • Launch the approved remote tool (AnyDesk/TeamViewer). Share the one-time session code and password.
  • Start a screen recording of the session for auditing purposes.
  • Monitor the vendor’s actions in real-time to ensure compliance with the approved scope.
Step 3: Terminate Session and Audit
  • Close the remote session immediately after task completion.
  • Disable the vendor’s administrative account.
  • Log session details in the Server Audit Register (Start/end time, actions performed, anomalies).

Outcome: The vendor completes the required tasks securely, with a full audit trail and no persistent access.

11. GitLab Administration (Repository Maintenance)

Scenario: A research team's code repository is bloated with large temporary data files, slowing down git clone operations.

Step 1: Identify Large Files in Repository
  • Log into the GitLab Web Interface → target repository → Settings → Usage Quotas.
  • Review the repository’s storage breakdown and identify the largest files (e.g., `.zip`, `.dat`).
Step 2: Remove Large Files from Git History
  • Use a history-rewriting tool such as BFG Repo-Cleaner or `git filter-branch` to purge the identified large files.
  • Example: `bfg --delete-files '.zip' my-repo.git`
  • Ensure that all branches and tags are updated and pushed to the remote repository.
Step 3: Perform Repository Garbage Collection
  • Access the GitLab server via SSH. Navigate to the repository directory.
  • Run Git’s garbage collection: `git gc --prune=now` and `git repack -ad`.
  • Verify repository size reduction using `du -sh` or GitLab’s usage stats.
Step 4: Prevent Future Repository Bloat
  • Update the repository `.gitignore` file to prevent accidental commits of large files.
  • Configure GitLab push rules to block large files from being committed.

Outcome: Repository size is significantly reduced, improving clone/fetch performance. Guidelines and push rules prevent future recurrence of repository bloat.
